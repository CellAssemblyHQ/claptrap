# Artifacts

An artifact is a derived representation of an entry, produced after the
original content has been fetched and transformed. Where an entry records
that a piece of content exists and what Claptrap knows about it, an
artifact records what that content actually looks like in a particular
output format.

The most common artifacts are extracted Markdown or HTML renderings of
an article, but the model is general. Any future format that the
extractor subsystem learns to produce, such as a PDF or a plain text
transcript, is represented as another artifact attached to the same
entry.

## Relationship to entries

Every artifact belongs to exactly one entry through `entry_id`. An entry
may have many artifacts, but it may only have one artifact per format.
The database enforces this with a unique index on the pair
`(entry_id, format)`, and the Catalog API upserts on that pair so that
re-extracting an entry replaces the previous artifact in place rather
than accumulating duplicates.

This shape means that asking "what is the Markdown body for entry X" is
always a single deterministic lookup. There is no notion of artifact
history or versioning in the schema today. If extraction runs again the
new payload simply overwrites the old one.

## Resource shape

Every artifact has the following fields.

| Field | Type | Description |
| --- | --- | --- |
| `id` | uuid | Primary key, generated by the database |
| `entry_id` | uuid | Foreign key into `entries` |
| `format` | string | Output format, one of `markdown`, `html`, `pdf` |
| `content` | string | The extracted payload itself |
| `content_type` | string | MIME type of the payload, for example `text/markdown` |
| `byte_size` | integer | Size of `content` in bytes, set by the pipeline at insert time |
| `extractor` | string | Name of the adapter that produced the artifact, for example `firecrawl` |
| `metadata` | map | Arbitrary structured metadata returned by the extractor |
| `inserted_at` | timestamp | When this row was first written |
| `updated_at` | timestamp | When this row was last replaced by a re-extraction |

The `extractor` field is purely informational. It records which adapter
produced the payload so that operators can trace provenance and so that
later code can decide whether a previously cached artifact is still
acceptable.

## How artifacts are produced

Artifacts are written almost exclusively by the extractor subsystem.
When the consumer subsystem ingests new entries it broadcasts an
`entries_ingested` event on the internal PubSub bus. The
`Claptrap.Extractor.Router` receives that event, filters for entries
that have a usable URL, and starts a supervised task per entry. Each
task runs the extractor pipeline, which calls a configured adapter for
every requested format and writes the result as an artifact through
`Claptrap.Catalog.create_artifact/1`.

See [the extractor architecture](../architecture/extractor.md) for the
full flow, including retry and failure semantics.

## How artifacts are consumed

Artifacts are read by anything downstream of extraction that needs the
rendered body of a piece of content. The HTTP API exposes them under
`/api/v1/artifacts` for direct inspection and listing, and producer
adapters that need the body of an entry rather than just its metadata
will typically join through `entry_id` to find the right artifact for
their preferred format.

Because the `(entry_id, format)` constraint makes lookup deterministic,
consumers should ask for a specific format rather than scanning. A
producer that wants Markdown should query for the Markdown artifact and
treat its absence as a signal that extraction has not yet completed for
that entry.

## Format constraints

The `format` field is validated at the schema level against a small
allowlist. Today that allowlist is `markdown`, `html`, and `pdf`.
Adding a new format requires three coordinated changes. The schema
validator must accept it, an adapter that supports it must exist, and
the extraction configuration must map it to that adapter. The format
string is the join key across all three.

## Validation rules

The artifact changeset requires `entry_id`, `format`, and `extractor`,
and rejects any `format` outside the allowlist above. The `extractor`
field must be a non-empty string. The foreign key into `entries` is
enforced at the database level, so deleting an entry without first
deleting its artifacts will fail rather than silently orphan rows.

`content` is not required by the changeset. The pipeline always writes
content on success, but the schema permits artifact rows without
payload bytes to leave room for future adapters that store content out
of band, for example by writing the body to blob storage and recording
only metadata in the catalog row.
